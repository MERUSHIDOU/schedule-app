# GitHub Issue テンプレート集

このファイルには、様々な種類のIssueを作成する際に使用できるテンプレートが含まれています。

MCPツール（`github:create_issue`）を使用してIssueを作成する際、以下のテンプレートから適切なものを選択し、`body`パラメータに使用してください。

---

## バグレポートテンプレート

```
## 概要
[バグの簡潔な説明]

## 再現手順
1. [最初のステップ]
2. [次のステップ]
3. [具体的な手順を記載]

## 期待される動作
[本来どのように動作すべきか]

## 実際の動作
[実際に何が起きるか]

## 環境
- OS: [Windows/Mac/Linux]
- ブラウザ: [Chrome/Firefox/Safari等]
- バージョン: [関連するバージョン情報]

## 追加情報
[スクリーンショット、エラーメッセージ、その他の情報]
```

### バグレポートの使用例

**MCPツール呼び出し**:
```
github:create_issue
{
  "owner": "MERUSHIDOU",
  "repo": "schedule-app",
  "title": "ログイン後にホーム画面が表示されない",
  "body": "## 概要\nログイン成功後、ホーム画面ではなく空白ページが表示される\n\n## 再現手順\n1. ログインページにアクセス\n2. 正しいメールアドレスとパスワードを入力\n3. ログインボタンをクリック\n\n## 期待される動作\nログイン後、ホーム画面（ダッシュボード）が表示される\n\n## 実際の動作\n空白ページが表示され、何も操作できない\n\n## 環境\n- OS: macOS Sonoma 14.2\n- ブラウザ: Chrome 120.0.6099.129\n- バージョン: v1.2.3\n\n## 追加情報\nコンソールエラー: 'Cannot read property user of undefined'",
  "labels": ["bug"]
}
```

## 機能リクエストテンプレート

```
## 概要
[どのような機能や機能が必要か]

## 解決する問題
[なぜこの機能が必要なのか、背景を説明]

## 提案する解決策
[どのように機能すべきか説明]

## 検討した代替案
[他に考えたアプローチがあれば記載]

## 受け入れ条件
- [ ] [条件1]
- [ ] [条件2]
- [ ] [条件3]

## 追加情報
[モックアップ、参考資料、関連リンクなど]
```

### 機能リクエストの使用例

**MCPツール呼び出し**:
```
github:create_issue
{
  "owner": "MERUSHIDOU",
  "repo": "schedule-app",
  "title": "ダークモード対応",
  "body": "## 概要\nアプリケーション全体でダークモードをサポートする機能\n\n## 解決する問題\n夜間や暗い環境でアプリを使用する際、明るい画面が目に負担をかける。多くのユーザーがダークモードを求めている。\n\n## 提案する解決策\n- 設定画面にダークモード切り替えオプションを追加\n- システム設定（OS）と連動したテーマ自動切り替え\n- すべての画面でダークテーマをサポート\n\n## 検討した代替案\n- ブラウザの拡張機能に任せる → ユーザー体験が統一されない\n- 一部の画面のみ対応 → 中途半端な実装になる\n\n## 受け入れ条件\n- [ ] ダークモードの切り替えUIを実装\n- [ ] すべてのコンポーネントでダークテーマ対応\n- [ ] システム設定との連動機能\n- [ ] テーマ設定の永続化\n\n## 追加情報\n参考: Material Design のダークテーマガイドライン\nhttps://m3.material.io/styles/color/dark-mode/overview",
  "labels": ["feature"]
}
```

## 改善テンプレート

```
## 概要
[何を改善すべきか]

## 現在の動作
[現在どのように動作しているか]

## 提案する改善
[どのように改善すべきか]

## メリット
[この改善がもたらす価値]

## 実装の考慮事項
[技術的な制約や考慮すべき点]
```

### 改善の使用例

**MCPツール呼び出し**:
```
github:create_issue
{
  "owner": "MERUSHIDOU",
  "repo": "schedule-app",
  "title": "スケジュール作成フォームのUX改善",
  "body": "## 概要\nスケジュール作成時の入力フローをより直感的にする\n\n## 現在の動作\n- すべてのフィールドを一度に入力する必要がある\n- 開始時刻を入力しても終了時刻は自動設定されない\n- 日付選択がカレンダーピッカーのみ\n\n## 提案する改善\n- 開始時刻を選択したら、1時間後を終了時刻としてデフォルト設定\n- 「今日」「明日」のクイックボタンを追加\n- フォーム入力のステップ分割（日時 → 詳細 → 確認）\n\n## メリット\n- 入力時間の短縮\n- ユーザーエラーの減少\n- より快適な操作体験\n\n## 実装の考慮事項\n- モバイル表示でのレイアウト調整\n- 既存のデータ構造への影響はなし\n- アクセシビリティの維持",
  "labels": ["improvement", "ux"]
}
```

```

### ドキュメント改善の使用例

**MCPツール呼び出し**:
```
github:create_issue
{
  "owner": "MERUSHIDOU",
  "repo": "schedule-app",
  "title": "開発環境セットアップガイドの追加",
  "body": "## 概要\nREADME.mdに開発環境のセットアップ手順を追加する\n\n## 現在の問題\n- 新しい開発者がプロジェクトに参加する際、環境構築に時間がかかる\n- セットアップ手順が口頭で伝えられている\n- 必要なツールやバージョン情報が明文化されていない\n\n## 提案する改善\n以下の情報を含むセクションをREADME.mdに追加：\n- 必要な前提条件（Node.js、npm等のバージョン）\n- インストール手順（ステップバイステップ）\n- 開発サーバーの起動方法\n- よくある問題とその解決方法\n\n## 対象読者\n- 新規参加の開発者\n- コントリビューター",
  "labels": ["documentation"]
}
```

## タスク/チェックリストテンプレート

```
## 概要
[タスクの説明]

## タスク一覧
- [ ] [タスク1]
- [ ] [タスク2]
- [ ] [タスク3]

### タスクの使用例

**MCPツール呼び出し**:
```
github:create_issue
{
  "owner": "MERUSHIDOU",
  "repo": "schedule-app",
  "title": "v2.0リリース準備",
  "body": "## 概要\nバージョン2.0のリリースに向けた準備タスク\n\n## タスク一覧\n- [ ] 機能凍結（1/15）\n- [ ] すべてのテストが通ることを確認\n- [ ] ドキュメントの更新\n- [ ] CHANGELOG.mdの作成\n- [ ] リリースノートの作成\n- [ ] ステージング環境でのテスト\n- [ ] 本番環境へのデプロイ\n\n## 完了条件\nv2.0が本番環境に正常にデプロイされ、すべての機能が動作している\n\n## 期限\n2026年1月31日",
  "labels": ["release"]
}
```

## ラベルの使用ガイド

### 種別ラベル
- `bug` - バグ報告
- `feature` - 新機能の追加（ユーザー向けの新しい機能）
- `improvement` - 改善（開発環境の向上、UX改善、パフォーマンス改善など）
- `documentation` - ドキュメント関連
- `refactoring` - リファクタリング

### 優先度ラベル
- `priority:critical` - 緊急対応が必要
- `priority:high` - 高優先度
- `priority:medium` - 中優先度
- `priority:low` - 低優先度

### 状態ラベル
- `blocked` - ブロックされている
- `in-progress` - 作業中
- `review` - レビュー待ち

### その他
- `good first issue` - 初めてのコントリビューター向け
- `help wanted` - 外部からの協力を求める
- `question` - 質問や議論

## MCPツール使用時のTips

### 1. 複数ラベルの指定
```json
"labels": ["bug", "priority:high", "blocked"]
```

### 2. 複数担当者の指定
```json
"assignees": ["developer1", "developer2"]
```

### 3. Issue本文での改行
本文中で改行を含める場合は`\n`を使用：
```json
"body": "## 概要\n説明文\n\n## 詳細\n詳しい内容"
```

### 4. Issue番号の参照
本文中で他のIssueを参照：
```json
"body": "このIssueは #123 に関連しています。\n\nCloses #456"
```

### 5. タイトルは簡潔に
- 50-70文字以内で要点を伝える
- 具体的な内容を含める
- 悪い例: "バグ修正"
- 良い例: "APIが無効なトークンで500エラーを返す"

---

## テンプレートのカスタマイズ

プロジェクトに合わせてテンプレートをカスタマイズできます：
- セクションの追加・削除
- 必須項目の追加
- プロジェクト固有の情報の追加

定期的にチームでテンプレートをレビューし、改善していくことをお勧めします。
