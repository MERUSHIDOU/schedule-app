# GitHub Issue テンプレート集

このファイルには、様々な種類のIssueを作成する際に使用できるテンプレートが含まれています。

## バグレポートテンプレート

```
## 概要
[バグの簡潔な説明]

## 再現手順
1. [最初のステップ]
2. [次のステップ]
3. [具体的な手順を記載]

## 期待される動作
[本来どのように動作すべきか]

## 実際の動作
[実際に何が起きるか]

## 環境
- OS: [Windows/Mac/Linux]
- ブラウザ: [Chrome/Firefox/Safari等]
- バージョン: [関連するバージョン情報]

## 追加情報
[スクリーンショット、エラーメッセージ、その他の情報]
```

### バグレポートの使用例

```bash
gh issue create \
  --title "ログイン後にホーム画面が表示されない" \
  --body "## 概要
ログイン成功後、ホーム画面ではなく空白ページが表示される

## 再現手順
1. ログインページにアクセス
2. 正しいメールアドレスとパスワードを入力
3. ログインボタンをクリック

## 期待される動作
ログイン後、ホーム画面（ダッシュボード）が表示される

## 実際の動作
空白ページが表示され、何も操作できない

## 環境
- OS: macOS Sonoma 14.2
- ブラウザ: Chrome 120.0.6099.129
- バージョン: v1.2.3

## 追加情報
コンソールエラー: 'Cannot read property user of undefined'" \
  --label "bug"
```

## 機能リクエストテンプレート

```
## 概要
[どのような機能や機能が必要か]

## 解決する問題
[なぜこの機能が必要なのか、背景を説明]

## 提案する解決策
[どのように機能すべきか説明]

## 検討した代替案
[他に考えたアプローチがあれば記載]

## 受け入れ条件
- [ ] [条件1]
- [ ] [条件2]
- [ ] [条件3]

## 追加情報
[モックアップ、参考資料、関連リンクなど]
```

### 機能リクエストの使用例

```bash
gh issue create \
  --title "ダークモード対応" \
  --body "## 概要
アプリケーション全体でダークモードをサポートする機能

## 解決する問題
夜間や暗い環境でアプリを使用する際、明るい画面が目に負担をかける。多くのユーザーがダークモードを求めている。

## 提案する解決策
- 設定画面にダークモード切り替えオプションを追加
- システム設定（OS）と連動したテーマ自動切り替え
- すべての画面でダークテーマをサポート

## 検討した代替案
- ブラウザの拡張機能に任せる → ユーザー体験が統一されない
- 一部の画面のみ対応 → 中途半端な実装になる

## 受け入れ条件
- [ ] ダークモードの切り替えUIを実装
- [ ] すべてのコンポーネントでダークテーマ対応
- [ ] システム設定との連動機能
- [ ] テーマ設定の永続化

## 追加情報
参考: Material Design のダークテーマガイドライン
https://m3.material.io/styles/color/dark-mode/overview" \
  --label "enhancement"
```

## 改善テンプレート

```
## 概要
[何を改善すべきか]

## 現在の動作
[現在どのように動作しているか]

## 提案する改善
[どのように改善すべきか]

## メリット
[この改善がもたらす価値]

## 実装の考慮事項
[技術的な制約や考慮すべき点]
```

### 改善の使用例

```bash
gh issue create \
  --title "スケジュール作成フォームのUX改善" \
  --body "## 概要
スケジュール作成時の入力フローをより直感的にする

## 現在の動作
- すべてのフィールドを一度に入力する必要がある
- 開始時刻を入力しても終了時刻は自動設定されない
- 日付選択がカレンダーピッカーのみ

## 提案する改善
- 開始時刻を選択したら、1時間後を終了時刻としてデフォルト設定
- 「今日」「明日」のクイックボタンを追加
- フォーム入力のステップ分割（日時 → 詳細 → 確認）

## メリット
- 入力時間の短縮
- ユーザーエラーの減少
- より快適な操作体験

## 実装の考慮事項
- モバイル表示でのレイアウト調整
- 既存のデータ構造への影響はなし
- アクセシビリティの維持" \
  --label "enhancement,ux"
```

## ドキュメント改善テンプレート

```
## 概要
[どのドキュメントを改善すべきか]

## 現在の問題
[現在のドキュメントの何が不十分か]

[どのように改善すべきか]

```

### ドキュメント改善の使用例

```bash
gh issue create \
  --title "開発環境セットアップガイドの追加" \
  --body "## 概要
README.mdに開発環境のセットアップ手順を追加する

## 現在の問題
- 新しい開発者がプロジェクトに参加する際、環境構築に時間がかかる
- セットアップ手順が口頭で伝えられている
- 必要なツールやバージョン情報が明文化されていない

## 提案する改善
以下の情報を含むセクションをREADME.mdに追加：
- 必要な前提条件（Node.js、npm等のバージョン）
- インストール手順（ステップバイステップ）
- 開発サーバーの起動方法
- よくある問題とその解決方法

## 対象読者
- 新規参加の開発者
- コントリビューター" \
  --label "documentation"
```

## ラベルの使用ガイド

### 種別ラベル
- `bug` - バグ報告
- `enhancement` - 新機能や機能改善
- `documentation` - ドキュメント関連
- `refactoring` - リファクタリング

### 優先度ラベル
- `priority:critical` - 緊急対応が必要
- `priority:high` - 高優先度
- `priority:medium` - 中優先度
- `priority:low` - 低優先度

### 状態ラベル
- `blocked` - ブロックされている
- `in-progress` - 作業中
- `review` - レビュー待ち

### その他
- `good first issue` - 初めてのコントリビューター向け
- `help wanted` - 外部からの協力を求める
- `question` - 質問や議論

## Tips

1. **タイトルは簡潔に**: 50-70文字以内で要点を伝える
2. **具体的に**: 曖昧な表現を避け、具体的な情報を提供
3. **テンプレートをカスタマイズ**: プロジェクトに合わせて調整
4. **関連情報をリンク**: 関連するIssueやPRを参照
5. **定期的な更新**: 進捗があればIssueをアップデート
