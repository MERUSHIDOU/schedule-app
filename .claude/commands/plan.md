# Planコマンド

このコマンドは **planner** エージェントを起動し、コードを書く前に包括的な実装計画を策定します。

## このコマンドの機能

1. **要件の再確認** - 何を構築すべきか明確化
2. **リスク特定** - 潜在的な問題とブロッカーを浮き彫りに
3. **段階計画の作成** - 実装をフェーズに分割
4. **確認待機** - 進む前にユーザーの承認が必須

## 使用する場合

以下の場合に `/plan` を使用：
- 新機能の開始
- 建築的な大きな変更
- 複雑なリファクタリング作業
- 複数のファイル/コンポーネントが影響を受ける
- 要件が不明確または曖昧

## 仕組み

plannerエージェントが以下を実行：

1. **リクエストを分析** し、要件を明確に言い直す
2. **フェーズに分割** し、具体的で実行可能なステップを提示
3. **コンポーネント間の依存関係** を特定
4. **リスクを評価** し、潜在的なブロッカーを特定
5. **複雑度を推定** (High/Medium/Low)
6. **計画を提示** し、明示的な確認を待機

## 使用例

```
ユーザー: /plan マーケット解決時のリアルタイム通知機能を追加したい

エージェント（planner）:
# 実装計画: リアルタイムマーケット解決通知

## 要件の再確認
- ユーザーが監視しているマーケットが解決した際に通知を送信
- 複数の通知チャネルに対応（アプリ内、メール、Webhook）
- 通知の確実な配信を確保
- マーケット結果とユーザーのポジション結果を含める

## 実装フェーズ

### Phase 1: データベーススキーマ
- 通知テーブルを追加（カラム: id, user_id, market_id, type, status, created_at）
- ユーザー通知設定テーブルをチャネル設定用に追加
- パフォーマンス用にuser_idとmarket_idにインデックスを作成

### Phase 2: 通知サービス
- lib/notifications.tsに通知サービスを作成
- BullMQ/Redisを使用した通知キューを実装
- 失敗した配信用に再試行ロジックを追加
- 通知テンプレートを作成

### Phase 3: 統合ポイント
- マーケット解決ロジックにフック（ステータスが「解決済み」に変更時）
- マーケット内のポジションを持つすべてのユーザーをクエリ
- 各ユーザーの通知をキューに登録

### Phase 4: フロントエンドコンポーネント
- ヘッダーにNotificationBellコンポーネントを作成
- NotificationListモーダルを追加
- Supabaseサブスクリプション経由でリアルタイム更新を実装
- 通知設定ページを追加

## 依存関係
- Redis（キュー用）
- メールサービス（SendGrid/Resend）
- Supabaseリアルタイムサブスクリプション

## リスク
- HIGH: メール配信可能性（SPF/DKIM必須）
- MEDIUM: マーケット当たり1000+ユーザーのパフォーマンス
- MEDIUM: マーケットが頻繁に解決する場合の通知スパム
- LOW: リアルタイムサブスクリプションのオーバーヘッド

## 推定複雑度: MEDIUM
- バックエンド: 4-6時間
- フロントエンド: 3-4時間
- テスト: 2-3時間
- 合計: 9-13時間

**確認待機中**: この計画で進める？ (yes/no/modify)
```

## 重要な注記

**重大**: plannerエージェントは、"yes" や "proceed" などの明示的な確認を受け取るまで、**コードを書きません**。

変更を希望する場合は、以下で回答：
- "modify: [変更内容]"
- "different approach: [代替案]"
- "skip phase 2 and do phase 3 first" (phase 2をスキップしてphase 3から開始)

## 他のコマンドとの統合

計画策定後：
- `/tdd` を使用してテスト駆動開発で実装
- ビルドエラーが発生した場合は `/build-fix` を使用
- 実装完了後 `/code-review` を使用してレビュー
